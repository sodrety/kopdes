# Use a fixed Officer hierarchy for request approval

Accepted. This decision supersedes ADR 0005's assignable `admin` role and the routine post-approval Loan Start Date correction allowed by ADR 0011. Every User is linked to a Member, and a Member may additionally hold one fixed Officer Role: Manager, Ketua I, Ketua II, or Ketua Utama. The same User enters the Member Area by default and, when appointed as an Officer, explicitly switches to the permitted Admin Area without changing identity or starting another session. Every Loan Request and Penarikan follows the same sequential chain for every amount: Manager, Ketua I, Ketua II, then Ketua Utama. Rejection at any stage and Member cancellation are terminal; only Ketua Utama Final Approval creates the Loan or withdrawal Saving Record and confirms the externally completed activity required by ADR 0004.

Approval authority does not imply inherited operational access. Manager retains day-to-day back-office operations and the Manager approval stage. Ketua I and Ketua II receive read-only oversight plus their own approval stages. Ketua Utama receives the same oversight, Final Approval, and exclusive Officer-account administration. Page, API, action, and navigation access use one explicit fixed permission matrix. Multiple Officers may hold a role, and the same person may decide multiple stages only when they hold each current role at decision time. An Officer may currently decide a stage for their own Loan Request or Penarikan; conflict-of-interest enforcement is deferred.

An Officer Appointment changes only Admin Area authority. Officers retain the same Member Area eligibility, limits, and request rights as other Members, with no preferential financial treatment. Their Loan Requests and Penarikan follow the same full Approval Chain.

Requests remain immutable after submission. Manager fixes Proposed Loan Terms during the first approval stage; later Officers approve that snapshot or reject it. Approval History preserves immutable snapshots of the deciding Member ID and number, display name, Officer Role, authenticating User ID, decision, time, and optional approval note or mandatory rejection reason. Later Member profile, User credential, or Officer Appointment changes do not rewrite those snapshots. Officers see full history; Members see the current state and latest completed decision without Officer identity. Post-approval Loan Start Date correction is removed from routine operations.

Pending Penarikan reserves Available Withdrawal Balance without changing Saving Balance. Rejection or Member cancellation releases the reservation; Final Approval converts it into a withdrawal Saving Record. Direct routine creation of Sukarela withdrawal records is prohibited, and only linked Member users may submit Penarikan for now.

Ketua Utama appoints existing Members as Officers, deactivates or reactivates Officer Appointments, resets credentials, and changes Officer Roles. Each Member has at most one Officer Appointment lifecycle record, carrying only its current Officer Role and active status; Member name and number remain sourced from Member, while login email remains sourced from User. Creation, role changes, suspension, and reactivation append immutable audit events rather than creating additional appointments. User remains only the Member's shared login identity. An appointment reuses the Member's linked User or creates that Member's first User when none exists; it never creates a standalone Officer identity. Deactivating an Officer Appointment removes Admin Area authority while preserving the Member's status, User, Member Area, savings, loans, and request rights. Deactivating membership immediately suspends any active Officer Appointment; reactivating membership does not restore officer authority until Ketua Utama deliberately reactivates the appointment. An Officer credential reset acts on the Member's single shared User, invalidates all Member and Admin Area sessions, and requires a password change at next login without ending membership or the Officer Appointment. Routine appointment changes and membership deactivation must preserve at least one active Ketua Utama; a successor must exist before the last active Ketua Utama can be demoted, suspended, or made an inactive Member. Officer Role, appointment-status, membership-status, and credential changes invalidate existing sessions immediately. Approval decisions retain the deciding Officer and Officer Role snapshot even after later appointment changes. Officer administration changes are immutable audit events without password values.

Persistent in-app notifications are created when a request enters a role's stage and when a request reaches a terminal outcome. Each notification carries a Member or Officer audience: personal outcomes appear only in the Member Area, while actionable approval tasks appear only in the Admin Area. Notifications are per-user, read/unread, linked to the request, and resolved for other Officers when one Officer decides the stage. Creating, reactivating, or changing an Officer Appointment backfills notifications for requests already waiting at its assigned role; changing or suspending an appointment resolves notifications that are no longer actionable. These notification changes do not alter Approval History. Notification events are stored independently from delivery so a future email channel can consume the same events without coupling email to approval logic.

**Migration**

- Existing `admin` users become Manager Officers only after each User is explicitly mapped to a pre-existing Member.
- Migration fails when any legacy Officer User lacks an explicit Member mapping; it never invents placeholder membership data.
- When the mapped Member already has a User, that Member User remains canonical and the legacy standalone Officer User becomes a non-authenticating Historical User Identity: credentials are revoked, sessions invalidated, and reactivation is prohibited while its identifier remains available to historical records.
- When the mapped Member has no User, the legacy Officer User is linked to that Member and retains its credentials.
- Existing approval decisions and Officer audit events retain their original authenticating User IDs and are enriched with immutable Member identity snapshots from the explicit mapping.
- Migration fails when any historical Officer actor cannot be mapped to a Member.
- Existing pending Loan Requests and Penarikan begin at the Manager stage with empty Approval History.
- Existing terminal requests and financial records remain historical and are not retroactively given approval stages.
- The initial Ketua Utama is appointed through an existing `KETUA_UTAMA_MEMBER_ID`; bootstrap credentials create that Member's first User only when none exists.
- When the bootstrap Member already has a User, bootstrap preserves the existing email and password and only creates or reactivates the Ketua Utama appointment.

**Consequences**

- Role identifiers and approval stages are fixed stable English values; browser copy and status labels remain translated in English and Bahasa Indonesia.
- Member status, User availability, and Officer Appointment status require separate persistence and authorization checks.
- Approval decisions, Officer audit events, notifications, and withdrawal reservations require durable data with concurrency-safe single-stage transitions.
- A missing active Officer at any stage delays the chain; higher roles cannot skip that stage.
- Existing admin-only middleware, routes, templates, tests, migrations, seed data, and exports must move to capability-based authorization.
